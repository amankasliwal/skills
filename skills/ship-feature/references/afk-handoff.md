# AFK handoff: roles, classification, and the agent brief

How an issue becomes safe for an unattended agent to pick up, and what contract it works from.
This aligns with the ticketing (`to-tickets`) and `triage` skills — it doesn't replace them, it
just records how this pipeline leans on them.

## HITL vs AFK, and the triage roles they map to

The ticketing step marks every ticket (vertical slice) as **HITL** (needs human interaction — an
architectural decision, a design review, a judgment call) or **AFK** (can be implemented without
human interaction — the final merge is always the human's). **Prefer AFK where possible**; reach
for HITL only when human interaction is genuinely required.

Those ticket types map onto the canonical **triage roles** that live on the issue tracker:

| Ticket type | Triage role | What the pipeline does with it |
|---|---|---|
| AFK | `ready-for-agent` | A local worker subagent implements it as an open, stacked PR |
| HITL | `ready-for-human` | Surfaces it to the user; never auto-implements |

These are *canonical role names*. The real label/state strings in your tracker (Linear, Jira,
GitHub, …) come from your triage config if a setup step has been run; otherwise use the role names
directly. Don't invent new labels — duplicates break the conductor's `ready-for-agent` selection.

## When to keep something HITL

The governing question is **reversibility**, not difficulty: *if a fresh agent gets this wrong,
how cheap is it to notice and re-roll?* Keep an issue `ready-for-human` if any hold:

- The spec is ambiguous, or implementing it forces a product / UX / architecture decision.
- It does something hard to undo: a destructive or data-shaped migration, a secret rotation,
  anything touching prod data, billing, or auth/security.
- It's infra work the project routes through review (IaC, a new service, env-var wiring).
- Correctness can't be proven locally without human judgment.
- You're simply unsure. **Default to `ready-for-human`** — a minute of the user's attention beats
  an unattended worker confidently shipping the wrong irreversible thing.

## The agent brief is the contract

When an issue moves to `ready-for-agent`, triage posts an **agent brief** — the authoritative spec
the AFK agent works from. The issue body and discussion are context; the brief is the contract. It
must also name the issue's **base = its assigned stack branch** (the feature base for the first
worker, or a prior worker's branch) — branch from it and PR onto it, and **never merge**. The base
already carries the spec context + the prior stack, so a fresh worker inherits both. The
principles that make a brief survive days in the queue while the codebase changes underneath it:

- **Durability over precision** — describe interfaces, types, and behavioral contracts. Name the
  types / signatures / config shapes to look for. **Never** reference file paths or line numbers;
  they go stale.
- **Behavioral, not procedural** — say *what* the system should do, not *how* to implement it. The
  agent explores fresh and makes its own implementation calls.
- **Complete, testable acceptance criteria** — every brief needs concrete, independently verifiable
  criteria. They are the agent's definition of done.
- **Explicit scope boundaries** — state what is out of scope so the agent doesn't gold-plate.

### Brief template

```markdown
> *Generated by AI during triage.*

## Agent Brief
**Category:** bug / enhancement
**Summary:** one line
**Base branch:** <assigned stack branch — the feature base or a prior worker's branch> (branch from it, PR onto it — NOT the integration branch; do NOT merge)

**Current behavior:** what happens now (the status quo this builds on / the bug)
**Desired behavior:** what should happen after, including edge cases and error conditions

**Key interfaces:**
- `TypeName` — what changes and why
- `functionName()` — current vs desired return/behavior
- Config shape — any new options

**Acceptance criteria:**
- [ ] Specific, testable criterion
- [ ] …

**Out of scope:**
- Adjacent thing that must NOT change here
```

A `ready-for-human` issue uses the *same* structure, plus a note on why it can't be delegated
(judgment calls, external access, design decisions, manual testing).
