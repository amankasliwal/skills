# Worker orchestration (local, stacked, no-merge)

Phase A's launch step dispatches **local worker subagents** that implement the
`ready-for-agent` issues. They run **in this local session** — not on the cloud — because a
worker needs the local repo working tree, write permissions, and full context.

## The branching model — a STACK, not a merge-accumulator

**Naming:** `base`, `feat-1`, `feat-2` below are placeholders for *stack positions*, never
literal branch names — name every branch after its work, per your repo's convention. `<integration-branch>`
is wherever finished work merges (`main`, `develop`, …).

Nothing is merged until the whole feature is done and the human reviews it. Work accumulates as
a **stack of open PRs**, not via merges into a shared branch:

- **`base`** — created off `<integration-branch>` at kickoff; carries the spec / ADR / domain
  docs. Pushed, but **never merged** until the feature is complete and reviewed.
- **worker-1** bases on `base`, creates `feat-1`, opens a PR `feat-1 → base`, and **stops — it
  does NOT merge**.
- **worker-2 / worker-3** (parallel, if independent) base on `feat-1`, create their branches,
  open PRs onto `feat-1`, and stop. And so on up the stack.
- The human reviews the whole stack end-to-end and merges it bottom-up
  (`feat-N → … → feat-1 → base → <integration-branch>`). **The skill never merges anything.**

This is the whole point: `<integration-branch>` stays untouched until the feature is genuinely
done — no half-feature specs or slices land without a review cycle.

## Base-branch selection (how the stack is built)

The conductor (this session) builds the stack from the issues' dependency graph:

- An issue with **no open dependency** bases on `base` — or, to keep building on prior work, on
  the **current tip** of the stack.
- An issue **blocked by** another bases on that blocker's branch — it inherits the blocker's work
  directly (the base branch already contains it; no merge needed).
- **Mutually-independent** issues that share a base run **in parallel** as siblings off that base.
- **Never require combining two un-merged sibling branches.** If a later issue needs work from two
  parallel siblings, linearize them (base one on the other) so every issue always has exactly one
  base branch.

## Running workers locally (no cloud, no cron)

Dispatch each worker as a **subagent in this session** (the Agent tool) so it has the local repo,
write permissions, and full context. Run them in **dependency waves**: a wave's workers run in
parallel; the next wave starts once the branches it bases on exist.

- **Parallel workers need isolation.** Two subagents editing one working tree collide — give each
  parallel worker its **own git worktree** (Agent tool `isolation: "worktree"`, or the
  `superpowers:using-git-worktrees` skill), checked out on its base branch. Sequential workers can
  reuse a tree. (`superpowers:dispatching-parallel-agents` is the fan-out pattern.)
- Each worker gets: its issue + agent brief, its **base branch**, and the per-issue playbook. It
  returns its **branch name + PR link** — which becomes the base for whatever stacks on top. It
  must **not merge**.
- Orchestration is in-session and stateful (the conductor tracks the graph, the branches, and
  what's done), so there is **no cron, no claim/lock, no stop-debounce**.
- The conductor dispatches **only issues currently labelled `ready-for-agent`**, and **never
  force-relabels** an issue — so a human can pause one by removing the label and the conductor
  simply skips it.

## Worker subagent prompt (fill the `<…>` and dispatch)

```
You are the worker for issue <issue-id> in the "<feature-name>" pipeline, running locally in your
own git worktree checked out on <base-branch> (= the feature base or a prior worker's branch).

Follow the per-issue playbook from this issue's agent brief:
- git fetch; branch off <base-branch> (NOT the integration branch), named per the repo convention.
- Implement with the TDD skill (red → green → refactor); don't weaken/delete existing tests.
- BEFORE the PR: run a code-review subagent (Opus) vs <base-branch>; fix Critical + Major.
- Open the PR NON-INTERACTIVELY, base = <base-branch> (NOT the integration branch).
- Review loop (correctness + simplicity): code-review subagent + flag-over-engineering pass +
  apply-cleanups (then re-run tests) ⇄ superpowers:receiving-code-review (diff vs <base-branch>)
  until clean.
- E2E verify on local against the brief's acceptance criteria; capture evidence. If E2E cannot
  run here, stop-and-surface (re-label ready-for-human) — do NOT fake it.
- DO NOT MERGE. Move the issue to in-review, REMOVE the ready-for-agent label, leave a trail
  (PR link + evidence), and return your branch name + PR link.
Stop-and-surface (re-label ready-for-human, post what's blocking prefixed `> *Generated by AI.*`)
if: the spec turns ambiguous, the work needs an irreversible action (migration/secret/prod-data/
auth), the review loop won't converge, or E2E fails / can't run.
```

## When all workers are done — hand the stack to the human

Do NOT merge, and do NOT open `base → <integration-branch>` yet (`base` only has the spec; the
workers' code lives in the stacked PRs). Instead, report the **full stack** for review:

- the stack/tree — `base ← feat-1 ← {feat-2, feat-3} ← …` — with each PR link + its issue,
- the **bottom-up merge order** the human follows once they've reviewed + verified,
- any `ready-for-human` issues still waiting.

The human runs the end-to-end review + verification loop on the stack, merges it up, and only then
opens `base → <integration-branch>`. That final review/merge is theirs, by design — the skill
stops at "stack ready for your review."
